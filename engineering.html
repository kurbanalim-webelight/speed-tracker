01 — Architecture

System Architecture

Four-layer stack from physical sensing to mobile display. Each layer has one job and one interface.

INTERFACE 1
Radar → ESP32
UART Serial @ 115200 baud. GPIO16 (RX). Raw speed values as ASCII strings every ~50ms.
INTERFACE 2
ESP32 → Phone
BLE NOTIFY characteristic. JSON payload pushed to Flutter on every valid detection.
INTERFACE 3
Power Bank → All
Mi 20,000mAh → ESP32 via USB. ESP32 VIN (5V) → Radar VCC. Shared GND.
02 — Component

Radar Sensor — HLK-LD303

The "eye" of the system. Detects ball motion using the Doppler effect and streams speed values over UART.

Why HLK-LD303 Was Chosen
✓Direct UART output — no FFT or ADC math needed on ESP32. Speed arrives as a number.
✓3.3V UART logic — plugs directly into ESP32 GPIO16 without a level shifter.
✓5–8m range — exactly right for a box cricket pitch of 10–14m.
✓Compact tripod-mount — small enough to attach to any standard tripod head.
⚠️

Voltage Warning for Engineer: Always measure the UART TX voltage of the exact module received with a multimeter before wiring. If it reads >3.3V (some batches output 5V logic), install a BSS138 or TXS0102 logic level shifter between radar TX and ESP32 GPIO16. ESP32 GPIO inputs are NOT 5V tolerant — direct connection will damage the chip.

06 — Installation

Field Placement & Mounting

Placement is the single most important factor for accuracy. Wrong position = wrong speed. Follow these diagrams exactly.

View 1 — Top (Bird's Eye)
Radar placed 1–2m behind wicket, center-aligned with middle stump, antenna pointing directly at bowler.
View 2 — Side (Elevation)
Mount radar at stump height (65–75cm from ground) to align with the natural ball travel arc.
View 3 — Front (Bowler's Angle)
Radar center must align with middle stump. 0° horizontal tilt — no tilt up or down. Any angle creates cosine error.

07 — Firmware

ESP32 Firmware Logic

State machine with 5 states. Every ball triggers one full cycle — IDLE → TRACKING → PROCESSING → SENDING → COOLDOWN → IDLE.

Firmware Pseudocode
// ── Configuration ─────────────────────────────────────────────
const float MIN_SPEED   = 20.0;   // km/h — tracking start threshold
const float MAX_SPEED   = 200.0;  // km/h — noise ceiling
const float STOP_SPEED  = 10.0;   // km/h — tracking end threshold
const int   COOLDOWN_MS = 3000;  // ms  — post-delivery lockout
const int   MAX_TRACK   = 2000;  // ms  — max valid track window
const int   MIN_TRACK   = 100;   // ms  — min valid track window

// ── Main loop ─────────────────────────────────────────────────
bool    tracking   = false;
float   maxSpeed   = 0;
long    trackStart = 0;
long    cooldownEnd= 0;

loop() {
  if (millis() < cooldownEnd) return;  // in cooldown — skip

  float speed = readRadarUART();        // parse ASCII from Serial2

  if (speed > MAX_SPEED) return;        // noise ceiling

  if (!tracking && speed > MIN_SPEED) {
    tracking   = true;
    trackStart = millis();
    maxSpeed   = speed;
  } else if (tracking) {
    maxSpeed = max(maxSpeed, speed);   // keep highest

    long elapsed = millis() - trackStart;

    if (elapsed > MAX_TRACK) {         // too long = not a ball
      tracking = false;  maxSpeed = 0;  return;
    }
    if (speed < STOP_SPEED) {          // ball passed
      if (elapsed >= MIN_TRACK) {
        sendBLE(maxSpeed);               // ← valid delivery
        cooldownEnd = millis() + COOLDOWN_MS;
      }
      tracking = false;  maxSpeed = 0;
    }
  }
  delay(50);  // UART_POLL_MS
}

BLE Protocol

ESP32 acts as a BLE peripheral. Flutter is the central. Speed data flows via NOTIFY characteristic — no polling needed.

Critical: UUIDs must match byte-for-byte in both ESP32 firmware and Flutter app. A single character difference = devices cannot communicate. Never change one side without updating the other.

BLE Reconnection Behavior
On Disconnect
ESP32 detects onDisconnect callback. Immediately restarts BLE advertising. No physical reset required.
During Reconnect
Radar processing continues in background. No deliveries are missed. Speed buffer is maintained.
After Reconnect
Flutter reconnects, re-subscribes to NOTIFY. Next delivery appears normally. No restart needed on either device.
09 — End-to-End

Full Delivery Walkthrough

One ball — start to finish. Every microsecond of system behavior from ball release to screen update and reset.

Complete Timing Diagram — One Delivery
PHASE 1
System Ready
T = 0 (before bowl)
Power bank ON. ESP32 running. Radar scanning. BLE connected. Radar reads 0 km/h — no object. App shows "READY — Waiting for delivery".
PHASE 2
Bowler Releases Ball
T = 0 ms
Ball leaves bowler's hand. Ball is still far from radar detection zone. Radar still reads 0 km/h. App screen unchanged.
PHASE 3
Ball Enters Detection Zone
T ≈ 300 ms
Ball approaches radar. Reflected signal arrives. Radar streams: 45→89→112→124 km/h. ESP32: speed >20 → tracking STARTED. Buffer: [45,89,112,124].
PHASE 4
Ball Passes Radar
T ≈ 550 ms
Ball moves past sensor. Speed drops: 119→98→61→8 km/h. Speed <10 → tracking ENDED. Full buffer max = 124 km/h. Duration ~250ms → VALID.
PHASE 5
BLE Notification Sent
T ≈ 600 ms
ESP32 sends: {"speed":124,"unit":"km/h","delivery":1}. Cooldown timer starts immediately. Radar input now ignored for 3000ms.
PHASE 6
App Displays Speed
T ≈ 650 ms
Flutter receives BLE notification. UI updates: big "124 km/h" shown. Speed saved to delivery history. Previous speed moves to history list.
PHASE 7
Cooldown Period
T = 650ms → 3650ms
ESP32 ignores all radar input. Prevents bat swing / fielder movement from triggering a false delivery. App shows speed + optional countdown timer.
PHASE 8
Reset — Ready for Next Ball
T = 3650 ms
Cooldown ends. ESP32 back to IDLE. Radar active again. App shows "READY — Last Speed: 124 km/h". Bowler can deliver next ball.
10 — Reliability

Error Handling & Edge Cases

All known failure modes, what the system does automatically, and how to fix non-auto-handled cases.

ScenarioWhat HappensAuto-Handled?Fix / Action
Bat swing false trigger Bat at 35 km/h exceeds 20 km/h threshold — triggers tracking Partial Raise MIN_SPEED to 40 km/h during calibration. Bat swings rarely exceed 50 km/h.
BLE disconnect mid-match ESP32 detects disconnect event. BLE advertising restarts instantly. Radar keeps running. Auto Flutter auto-reconnects. No data lost. No reset needed on either device.
Ball hits radar directly Spike reading >200 km/h generated Auto MAX_SPEED ceiling discards it. Nothing sent to app. System resets after cooldown.
Power bank dead ESP32 shuts down. BLE disconnects. App shows "Device disconnected". Manual Charge power bank. Reconnect. No data recovery (history stored in app only).
Multiple people moving Multiple low-speed blips below 20 km/h Auto All filtered by MIN_SPEED = 20 km/h threshold. Walking speed ≈ 5 km/h.
Garbage UART data Malformed bytes from radar — toFloat() returns 0.0 Auto 0.0 < MIN_SPEED → ignored. If persistent: verify baud rate (115200) and logic level (3.3V).
11 — Calibration

Calibration Process

Run calibration before every new venue or after any hardware change. Target: ±2–3 km/h accuracy.

1
Physical Setup
Place radar at exact field position — behind batter wicket, facing bowler, at stump height (65–75cm). Secure tripod firmly.
2
Test Deliveries
Bowl 5–10 test deliveries at known pace. Use a consistent bowler for each calibration run. Record every result.
3
Compare
Compare readings against a reference — another speed gun, known-pace bowler, or published professional bowling averages.
4
Tune Thresholds
If bat swings trigger false readings, raise MIN_SPEED. If valid balls are missed, lower it. Adjust in firmware and reflash.
5
Repeat Testing
Bowl another 5–10 deliveries after each threshold change. Repeat until 5 consecutive readings are consistent within ±3 km/h.
6
Lock & Document
Record final MIN_SPEED, MAX_SPEED, and COOLDOWN_MS values in project config file. Tag the firmware version.
